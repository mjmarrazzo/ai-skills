# knowledge-capture — DESIGN

Status: v1. The on-disk format and write payload are a long-lived contract that five sibling skills depend on. Bumps to the schema version require a written migration path.

## Goal

A per-repo, gitignored knowledge file set that accumulates the small lessons a session is going to learn anyway — what bit us, how this codebase actually works, which tools have which quirks — so the NEXT session (or the next blueprint Phase 1) doesn't relearn them. Never silently. Always at the user's word.

Not in scope: a wiki, a search index, freeform notes, anything the user is supposed to maintain by hand. This is a structured log written by skills at well-defined checkpoints, with the user as the gate.

## The shape of the contract

Every sibling skill reads from or writes to a single per-repo directory:

```
.claude-knowledge/
├── README.md       # auto-generated index — one line per entry, sorted newest-first, superseded entries at bottom
├── gotchas.md      # "thing that bit me, what to do" — append-only
├── patterns.md     # "this is how X works in this repo" — append-only
├── stack-notes.md  # tooling quirks (gradle, mise, SAM, AWS profiles) — append-only
└── .schema.json    # schema version pin + per-kind stale thresholds + title charset rules
```

The directory is created on the **first WRITE** in a repo, not on first read. A first read against a missing dir returns an empty digest with no side effects. This matters because blueprint Phase 1 reads happen against fresh clones constantly; materializing an empty `.claude-knowledge/` on every fresh clone would be noise.

When the directory is created, `.claude-knowledge/` is appended to `.gitignore` (idempotently). The user opts in to committing by removing the line — that's the deliberate act.

## Entry format

Every entry across the three kind-files follows the same line-prefix-parseable shape:

```markdown
## 2026-05-14-gradle-daemon-hang-a1b2c3 — Gradle daemon hangs on M3 macs  [tags: gradle, m3, daemon]
**Context:** Running `./gradlew :app:build` after a `mise use java@21` switch left a daemon orphaned on the old JDK.
**Lesson:** `rm -rf ~/.gradle/daemon/<version>/` and re-run. Don't `pkill -f gradle` — it corrupts the lock file.
**Source:** session: `debug-loop-task-3`, files: [build.gradle.kts, gradle/wrapper/gradle-wrapper.properties], commit: `abc1234`
```

The line-prefix patterns are pinned (see `references/entry-format.md`). The separator between slug and title is an **em-dash** (U+2014), not a hyphen-minus. Titles MUST NOT contain `[` or `]` (reserved for the tag bracket).

### Slug

Format: `<YYYY-MM-DD>-<kebab(title)>-<6char-hash>`.

The hash is a 6-character hex slice over the entry body (Context + Lesson + Source string) at write time. Two writes with identical titles on the same day produce different slugs as long as the body differs — so "gradle daemon hang" can be captured twice without ambiguity. Slug-only references (in supersede chains) are unambiguous targets.

### Tags

Inline bracket on the heading line: `[tags: foo, bar, baz]`. ≤4 tags per entry. Tags are free-form strings; we don't normalize them (`gradle` vs `Gradle` both legal). The cap is bloat-defence, not taxonomy.

### Append-only + supersede

No in-place edits, ever. Corrections append a new entry whose body begins with:

```markdown
**Supersedes:** 2026-05-14-gradle-daemon-hang-a1b2c3
```

The superseded entry stays in the file untouched. The README index marks it `[superseded by <slug>]` and sorts it below current entries within its kind section. The stale-marker logic skips superseded entries (they're not stale — they're intentionally archived).

If the supersede target slug is missing or matches multiple entries, the skill refuses the write and asks the user to disambiguate. We do not guess.

## Read API

One operation: a markdown digest, generated by grep-parsing the kind-files. Callers either receive the digest inline (return to the calling LLM, which folds it into its own artifact) or pass an `output_path` and the skill writes there.

Output shape (full grammar in `references/read-api.md`):

```markdown
## Known about this repo (from .claude-knowledge/)

### Gotchas (3 of 12)
- **[2026-05-14] Gradle daemon hangs on M3 macs** — kill `~/.gradle/daemon/<v>/`. (tags: gradle, m3, daemon)
- ...

### Patterns (2 of 5)
- ...

### Stack notes (1 of 3)  [stale?: 1]
- **[2025-08-01] [stale?] SAM local on port 5000** — port conflicts with Docker Desktop. (tags: sam, docker)

### Superseded (hidden, see file for history)
```

Defaults: 20 entries total, 6 per kind, newest first, superseded entries excluded. Caller can pass `limit`, `since`, `kind` to override. Empty result emits NO section at all — blueprint Phase 1 just skips it rather than printing "Known about this repo: nothing yet."

Per-kind stale thresholds (pinned in `.schema.json`):

| Kind | Stale threshold | Why |
|---|---|---|
| gotcha | 12 months | gotchas about specific commits or tool versions age, but slowly |
| pattern | 12 months | how-things-work ages slowly — repos don't restructure every quarter |
| stack-note | 3 months | tooling moves fast; gradle 7 stack notes go stale once gradle 9 lands |

Stale is **advisory**. Entries are never deleted. A newer entry sharing ≥1 tag with an older entry also marks the older `[stale?]` (relative staleness). The blueprint Phase 1 digest surfaces `[stale?]` inline so the user can decide whether to act.

## Write API

Cross-skill writes pass a structured payload. ALL fields except `tags` are required. The skill rejects writes missing required fields and surfaces what's missing:

```yaml
caller: <skill-name>            # required — also used for cycle guard
kind: gotcha | pattern | stack-note   # required
proposed:
  title: <string>               # required, ≤80 chars, no `[` or `]`
  context: <string>             # required, one sentence — what was happening
  lesson: <string>              # required, ≤2 sentences — the takeaway
  tags: [<string>, ...]         # optional, ≤4 tags
source:                         # required block — without it, entries become contextless
  files: [<path>, ...]          # files touched at the moment of capture
  commit: <sha-or-null>         # git rev-parse HEAD if in a repo
  session_marker: <string>      # caller-supplied: "debug-loop-task-3", "execute-plan-task-7-end", etc.
```

The `source` block is the graveyard guard. Without it, a year-old entry that says "Yarn workspaces require root install before subdir tests" is a one-liner with no provenance and no way for a future LLM to validate whether it's still true. With `files` and `commit`, the future reader can navigate to the captured state.

The caller is responsible for collecting `source.files` (via `git diff --name-only` over the relevant range) and `source.commit` (via `git rev-parse HEAD`).

### Confirmation flow

The skill **NEVER writes silently**. Every persisted entry requires explicit user assent. The "always ask" rule is permanent for v1; do NOT add a future auto-write opt-in.

**Interactive mode (default):** the cross-skill writes are *queued* during the session and presented as ONE batched prompt at session end (typically by `finish-branch` at its pre-flight gate, or by whatever skill terminates the session). The batched prompt uses `TodoWrite` to surface "N pending captures" mid-session so the user knows what's coming, and `AskUserQuestion` at the gate:

```
I noted 3 things worth remembering during this session:
1. [debug-loop] Gradle daemon hangs on M3 macs — kill ~/.gradle/daemon
2. [execute-plan] Yarn workspaces require root install before subdir tests
3. [debug-loop] SAM local port conflict when Docker Desktop running

Save these? (Yes all / Choose each / Skip all / Edit)
```

Batching is the friction control. Three per-checkpoint prompts during one debug session would train the user to ignore them.

**Auto mode (opt-in):** the skill writes the proposed entries to `.claude-plans/<active>/open-questions.md` (or `./.claude-results/<ts>/open-questions.md` ad-hoc) as deferred reviewables. No prompt. The user reviews at end of run. Auto mode is selected by an explicit phrase ("go full auto", "autonomous", "no questions") or the caller param `mode=auto`.

**User-initiated capture:** phrases like "remember this", "save that for later", "worth knowing" trigger a free-form capture flow. The skill asks one question — "OK, what's the gotcha (or pattern or stack-note)?" — synthesizes the entry, and queues it the same way a sibling-skill write would be queued. Same batched gate at session end.

### Cycle guard

`caller=knowledge-capture` is a no-op. The skill logs a single-line error to stderr and returns without acting. No skill in this monorepo writes to knowledge-capture recursively; this guard exists for the case where a future skill chains through and accidentally lands back here.

## Composition

| Posture | Skill | Direction |
|---|---|---|
| Callees (skills knowledge-capture may invoke) | (none — leaf) | — |
| Callers (skills that may invoke knowledge-capture) | blueprint, pre-task-research | read |
| | debug-loop, execute-plan, finish-branch | write |
| Cycle guard | `caller=knowledge-capture` | log error, no-op |

**Read injection points:**

- `blueprint` Phase 1: reads all kinds (limit 20 total, 6 per kind), folds the digest into `handoff.md` under "Known about this repo (from .claude-knowledge/)". If empty, no section is emitted.
- `pre-task-research`: reads all kinds before fanning out external sources. Local knowledge is the first section of `research.md`.

**Write injection points (the contract is pinned — callers MUST supply the payload above):**

- `debug-loop`: at Phase 7 ("log decision") — when the fix is in a different module than the symptom, OR more than one hypothesis was needed before localization. Pass `kind: gotcha`, `proposed` synthesized from the loop's decision log.
- `execute-plan`: at end-of-plan handoff — for each task that finished with `BLOCKED` status during execution OR exceeded its time budget. Pass `kind: gotcha`, `proposed` from the blocked-task notes.
- `finish-branch`: at pre-flight (after the clean-tree check, before `gh pr create`) — single prompt: "anything new worth remembering before opening this PR?". Also the canonical place to flush the queued batched prompts from this session.

Graceful degradation: every caller checks the sibling-installed marker (`~/.claude/skills/knowledge-capture/SKILL.md` OR `~/.claude/plugins/cache/**/skills/knowledge-capture/SKILL.md`) before invoking. If absent, the caller proceeds without writing and mentions the missing sibling once.

## Active workspace and ad-hoc behaviour

`.claude-knowledge/` lives at the **repo root** (or cwd if not in a git repo), NOT inside `.claude-plans/<workspace>/`. Knowledge is per-repo, not per-workspace. The workspace-resolution algorithm from composition-skills `decisions.md` does not apply here — knowledge spans workspaces by design.

Auto-mode deferred entries DO land in the active workspace's `open-questions.md` (or `./.claude-results/<ts>/open-questions.md` ad-hoc), using the same resolution algorithm. That's a separate file with a separate lifecycle.

## MSP detection

When the repo is MSP-detected (per the triangulated check in composition-skills decisions.md), the skill biases `source.session_marker` to include the branch's ticket key when available. Example: `debug-loop-MSP-7032-task-3`. Otherwise no behaviour change. Knowledge-capture does NOT inject `MSP-<ticket>:` prefixes anywhere — that's a commit-message convention, not a knowledge-entry one.

## First-write bootstrap

The first write in a repo performs the following idempotent steps:

1. `mkdir -p .claude-knowledge/`
2. Write `.schema.json` (literal content shipped in `references/schema.json`).
3. Write empty `gotchas.md`, `patterns.md`, `stack-notes.md` with a one-line header (`# Gotchas`, etc.).
4. Write an empty `README.md` with the auto-index header and "no entries yet" placeholder.
5. If `.gitignore` does not list `.claude-knowledge/`, append the line. If `.gitignore` does not exist in a git repo, create one with just that line.
6. Then append the entry being written.

Each subsequent write is just steps 6 + a `README.md` regeneration step.

## README index regeneration

After every write, the skill regenerates `README.md` by re-grepping the three kind-files for entry headings and emitting:

```markdown
# Knowledge index
_Last updated: 2026-05-14T10:30:00Z_

## Gotchas
- **2026-05-14** [gradle-daemon-hang-a1b2c3] Gradle daemon hangs on M3 macs `[gradle, m3, daemon]`
- **2026-04-01** [yarn-workspaces-root-install-d4e5f6] Yarn workspaces require root install before subdir tests `[yarn, monorepo]` [superseded by 2026-05-01-yarn-workspaces-pnpm-migration-...]

## Patterns
- ...

## Stack notes
- ...
```

Sorted newest-first within each kind; superseded entries at the bottom of their section. This is plain markdown, regeneration is cheap, and it gives a human (or a fresh LLM) a one-page view without parsing the underlying files.

## Schema version pinning

`.schema.json` carries `"schema_version": 1`. The read API consults it on every read to confirm compatibility. If a future version bumps the schema, knowledge-capture refuses to read and prints a migration prompt — we do not silently downgrade or guess. The version-1 schema (literal content in `references/schema.json`) is the contract every other skill relies on.

Bumps require a written migration path. Deferred to dogfooding.

## Anti-patterns

- **Silent writes "to save the user a prompt."** The whole skill's value evaporates the moment a user finds an entry they didn't approve. The batched gate exists precisely so we ask less often, not so we ask never. If you're tempted to skip the prompt, the right move is to defer to the batched gate, not to bypass it.
- **In-place edits "to correct" an old entry.** Always supersede. The history is the data — a future LLM reading the file needs to see what we used to think to understand why we now think differently.
- **Skipping `source.files` / `source.commit` / `session_marker` to "look concise."** An entry without provenance is a graveyard one-liner. The future reader has no way to navigate back to the captured state. The skill rejects writes missing these — don't try to route around it.
- **Auto-fabricating an entry from session context** without the user's explicit phrase or a sibling-skill payload. We never invent knowledge. The LLM rephrasing a complaint as a gotcha is not the user saying "remember this."
- **Long entries.** Each entry is ≤6 lines (heading + 3 body fields ≤2 lines each). Anything longer is two entries or a wiki page, not a gotcha.
- **Treating `.claude-knowledge/` as project documentation.** It's gitignored by default for a reason. Users opt in to committing per-repo; we do not push them.
- **Mid-session per-checkpoint prompts.** Three prompts during one debug session is skip-fatigue. Queue and batch. The exception is user-initiated capture ("remember this") — that gets one immediate "what should I write?" prompt because the user already asked.
- **Reading the file with a markdown parser.** The pinned grep patterns in `references/entry-format.md` are the parsing contract. Future LLMs use them. Anything fancier breaks across renderers and is overkill for line-prefix data.
- **Materializing the directory on first READ.** Blueprint Phase 1 reads against fresh clones constantly. First-read on missing dir is an empty digest with zero side effects.
- **Promoting auto-mode writes past the deferred-review log.** Auto-mode means "log to `open-questions.md` and let the user review later," not "write to the kind-files without asking." The "always ask" rule is mode-agnostic for v1.
